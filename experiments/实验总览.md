# 实验总览

从单智能体开始，到现在卡在哪里，再到后面准备怎么做的完整总思路。

后面如果主线理解再变，这里优先更新。

## 1. 最开始想做什么

总目标一直没变：

**希望把机器人从简单导航能力，逐步扩展到多机器人复杂交互，最后能在 dense 场景里稳定协同。**

最早的直觉很自然：

- 先让单机器人学会基本导航
- 再把能力推广到多机器人
- 再继续往更复杂、更密集的多车交互场景训练

也就是说，最初一直默认：

**同一个 actor 可以随着课程逐步变强，最后同时兼顾普通导航和复杂协同。**

## 2. 第一阶段：单智能体和最早的多智能体探索

对应目录：

- `01_第一次尝试/`

这阶段主要是在回答基础问题：

- 单智能体 TD3 路线能不能跑通
- 多智能体共享策略能不能工作
- reward、状态输入、局部 critic 这些基础改法有没有意义

这一阶段更像“把系统搭起来”，不是最后的方法故事。

但它很重要，因为后面所有主线都建立在这里的基础上。

## 3. 第二阶段：课程学习成为正式主线

对应目录：

- `02_课程学习/`

后面主线收成了课程学习。

大体顺序是：

1. 先补稳单车
2. 再把能力接回普通多车
3. 最后推进到更复杂的 dense 多车交互

### 3.1 前两段是成功的

前两段结论比较清楚：

- 单车补稳有效
- 普通多车主线回接有效

这说明：

**从简单到复杂逐步训练，这个大方向本身是对的。**

### 3.2 在普通五车阶段形成了几个关键节点

课程学习后面形成了几个重要 actor：

- `5A`
  - 普通五车 actor
  - 更像当前普通导航主干
- `5D`
  - 带几何邻域 critic 的五车 actor
  - 没有完全超过 `5A`
  - 但它带了一些更强的交互基础

到这里为止，主线还是比较顺的。

## 4. 第三阶段：旧的 pure dense 五车主线为什么没走通

真正卡住的是更复杂的 dense 多车阶段。

我们先试过比较原始的五车 dense 课程，比如：

- `stage2_dense_gentle`
  - 比最原始 dense 稍微软一点的五车 dense 课程
- `stage2_dense_bridge`
  - 想在普通五车和强 dense 五车之间加一个过渡台阶

但这两条线结果都不理想。

看到的核心现象是：

- 同步五车中心强交互太难
- 共享 policy、局部观测、无通信时，车很容易一起冲向冲突区
- reward 修过以后也没有根本改观
- actor 继续更新后仍然容易退化

所以旧的 pure dense 五车主线，基本没有学顺。

这一点很关键，因为后面很多新故事，都是从这个失败点拆出来的。

## 5. 为什么后来转成 pair / three 渐进 dense 路线

因为直接上 pure dense 五车太硬，所以后面没有继续死磕，而是把 dense 交互拆开。

新想法变成：

- 不再从“同步五车强对称中心冲突”起步
- 先做五车环境中的非对称双车主冲突
- 再做五车环境中的弱三车交互

对应就是：

- `stage3_asym_pair_5`
- `stage3_asym_three_5`

这一步不是换题。

更准确地说：

**这是把原来学不动的 pure dense 五车问题，拆成更可学的渐进 dense 版本。**

## 6. 在这条新 dense 线上，我们得到了什么

### 6.1 先看 `PAIR`

历史 dense 路线保留为结论。当前分支已经实现冻结 `5D` 的小型 adapter/residual，等待修复口径下的 baseline 和训练验证。

训练第三轮达到 `0.921 / 0.079 / 0.750`，说明 `5D` 是合理的 dense warm start。

所以现在角色逐渐清楚了：

- `5A`
  - 普通 actor
- `5D`
  - 普通到 dense 的桥接 actor
- `PAIR(from_5d)`
  - 当前 dense 专门化基线；正式 dense 测试尚未超过 `5D`

### 6.2 benchmark 也要分层

为了避免后面越讲越乱，现在测试口径应该固定成三层：

- `standard_5`
  - 普通场景主测试
- `stage3_asym_three_5`
  - dense 场景主测试
- `stage2_dense`
  - 只作为压力测试，不作为当前主 benchmark

这里特别说明：

- `stage3_asym_three_5` 不是唯一 dense 场景
- 但它是当前这条主线里最合理、最稳定、最适合比较模型能力的 dense benchmark

## 7. 第四阶段：为什么 single actor overwrite 讲不通

虽然 `PAIR` 这条渐进 dense 线比旧 pure dense 顺得多，但后面又遇到另一个更本质的问题：

**继续训练单一 actor，会退化。**

当前 `PAIR(from_5d) -> THREE_5` 的三轮 full success 从 `0.625` 下降到 `0.438`、`0.375`。

反复出现的现象都很像：

- warm start 模型本身还能跑
- 但 actor 一继续更新，就容易掉
- 有时普通场景能力先掉
- 有时 dense 场景也没有继续变强

所以这里得到的核心判断是：

**继续覆盖式训练同一个 actor，并不是纯增益。**

也就是说，普通导航能力和 dense 协同能力之间，至少存在明显的训练冲突。

## 8. 当前方法判断

现有结果已经足以说明，继续覆盖训练单一 actor 会破坏已有能力，而现有双 Actor 又缺少互补性。当前方向是冻结 `5D`，只增加小型 adapter/residual。

当前主矛盾还是：

**actor 在新阶段继续更新时的稳定性问题。**

## 9. 第五阶段：为什么现在转到“保留专门化”

对应目录：

- `04_保留专门化/`

到这里，主线判断已经比较清楚：

1. pure dense 五车旧主线不顺
2. `PAIR / THREE_5` 这种渐进 dense 课程更合理
3. `5D -> PAIR` 是目前最健康的 dense 专门化链路
4. 但继续把同一个 actor 往更复杂 dense 场景硬训，仍然容易退化

所以现在最自然的思路不再是：

- 继续硬改同一个 actor

而是：

- 保留已经学好的普通 actor
- 保留 bridge actor
- 单独准备 dense 专门化 actor
- 然后只在强交互局部状态下做增强，而不是全局重写整个策略

这就是“保留专门化”现在的来源。

## 10. 目前主线该怎么讲

现在最顺的故事，不是“我们发明了两个 actor 切换”。

更顺的讲法是：

**我们先证明了直接继续训练单一 actor 不稳定；然后发现不同阶段得到的 actor 确实有不同能力偏向；因此不再追求把同一个 actor 硬训成全能，而是转向 preserve-and-specialize。**

具体角色是：

- `5A`
  - 保留普通导航能力
- `5D`
  - 普通到 dense 的桥接基线
- `PAIR(from_5d)`
  - 当前 dense 专门化基线

## 11. 当前真正优先的方向

最近导师给的反馈很关键，意思可以收成三句：

1. 注意力可以试，但不一定是主要提升来源
2. 双 Actor 门控依赖专家互补，但 `5A + 5D` 不满足这一条件
3. 当前方案改为冻结单个 `5D`，只训练小型 adapter/residual

这直接影响了当前主线优先级：

- 不是先赌 attention
- 而是先做 gating

也就是说，接下来主线应该讲成：

**冻结 `5D` Actor，不再覆盖完整策略；训练最小 adapter/residual，并与固定 `5D` 公平比较。**

## 12. 现在已经做过什么

到目前为止，已经完成的大块工作可以概括成：

- 单智能体和最早的多智能体基础探索
- 正式课程学习主线
- 普通五车 actor 主线建立（`5A` / `5D`）
- 旧 pure dense 五车主线失败验证
- `5D -> PAIR(from_5d)` dense 专门化
- `PAIR(from_5d) -> THREE_5` 的继续训练验证
- overwrite 路线退化验证
- 粗双 actor 切换尝试

这些工作现在都不是孤立的，它们一起支持了当前主线判断。

## 13. 最近完成了什么

刚补完的是：

- `PAIR(from_5d)` 的正式 test
  - `standard_5`：`0.891 / 0.085 / 0.573`
  - `stage3_asym_three_5`：`0.880 / 0.122 / 0.575`

结论也很直接：

- `PAIR(from_5d)` 在标准五车上仍保留了一定普通导航能力
- 但它在正式 dense benchmark 上没有超过 `5D`
- 所以它可以保留，但暂时不能直接定成门控里的 dense expert

## 14. 接下来具体要做什么

正式顺序已冻结在：

```text
experiments/04_保留专门化/05_论文主线/README.md
```

顺序固定为一条线：

1. 定义并实现 synchronized nominal conflict graph。
2. 实现带完整 manifest 的 interaction-density 程序化生成器。
3. 冻结 train/validation/test splits，重跑修复口径 generalist baseline。
4. 冻结 `5D`，训练 residual specialist。
5. 用 paired episodes 审计专家互补性；未达阈值则不训练 gate。
6. 互补性成立后，冻结专家并训练本地 temporal gate。
7. 完成三 seed、density sweep、held-out case、跨数量和跨地图实验。

## 15. 现在这一版最简短的一句话

**基础导航由冻结 `5D` 保留；下一步只验证小型 adapter/residual 能否改善 dense 交互且不破坏普通能力。**
