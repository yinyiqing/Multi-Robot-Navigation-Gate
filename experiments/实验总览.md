# 实验总览

从单智能体开始，到现在卡在哪里，再到后面准备怎么做的完整总思路。

后面如果主线理解再变，这里优先更新。

## 1. 最开始想做什么

总目标一直没变：

**希望把机器人从简单导航能力，逐步扩展到多机器人复杂交互，最后能在 dense 场景里稳定协同。**

最早的直觉很自然：

- 先让单机器人学会基本导航
- 再把能力推广到多机器人
- 再继续往更复杂、更密集的多车交互场景训练

也就是说，最初一直默认：

**同一个 actor 可以随着课程逐步变强，最后同时兼顾普通导航和复杂协同。**

## 2. 第一阶段：单智能体和最早的多智能体探索

对应目录：

- `01_第一次尝试/`

这阶段主要是在回答基础问题：

- 单智能体 TD3 路线能不能跑通
- 多智能体共享策略能不能工作
- reward、状态输入、局部 critic 这些基础改法有没有意义

这一阶段更像“把系统搭起来”，不是最后的方法故事。

但它很重要，因为后面所有主线都建立在这里的基础上。

## 3. 第二阶段：课程学习成为正式主线

对应目录：

- `02_课程学习/`

后面主线收成了课程学习。

大体顺序是：

1. 先补稳单车
2. 再把能力接回普通多车
3. 最后推进到更复杂的 dense 多车交互

### 3.1 前两段是成功的

前两段结论比较清楚：

- 单车补稳有效
- 普通多车主线回接有效

这说明：

**从简单到复杂逐步训练，这个大方向本身是对的。**

### 3.2 在普通五车阶段形成了几个关键节点

课程学习后面形成了几个重要 actor：

- `5A`
  - 普通五车 actor
  - 更像当前普通导航主干
- `5D`
  - 带几何邻域 critic 的五车 actor
  - 没有完全超过 `5A`
  - 但它带了一些更强的交互基础

到这里为止，主线还是比较顺的。

## 4. 第三阶段：旧的 pure dense 五车主线为什么没走通

真正卡住的是更复杂的 dense 多车阶段。

我们先试过比较原始的五车 dense 课程，比如：

- `stage2_dense_gentle`
  - 比最原始 dense 稍微软一点的五车 dense 课程
- `stage2_dense_bridge`
  - 想在普通五车和强 dense 五车之间加一个过渡台阶

但这两条线结果都不理想。

看到的核心现象是：

- 同步五车中心强交互太难
- 共享 policy、局部观测、无通信时，车很容易一起冲向冲突区
- reward 修过以后也没有根本改观
- actor 继续更新后仍然容易退化

所以旧的 pure dense 五车主线，基本没有学顺。

这一点很关键，因为后面很多新故事，都是从这个失败点拆出来的。

## 5. 为什么后来转成 pair / three 渐进 dense 路线

因为直接上 pure dense 五车太硬，所以后面没有继续死磕，而是把 dense 交互拆开。

新想法变成：

- 不再从“同步五车强对称中心冲突”起步
- 先做五车环境中的非对称双车主冲突
- 再做五车环境中的弱三车交互

对应就是：

- `stage3_asym_pair_5`
- `stage3_asym_three_5`

这一步不是换题。

更准确地说：

**这是把原来学不动的 pure dense 五车问题，拆成更可学的渐进 dense 版本。**

## 6. 在这条新 dense 线上，我们得到了什么

### 6.1 先看 `PAIR`

一开始的 dense actor 候选是：

- `PAIR(from_5a)`

后来又补做了更合理的一版：

- `PAIR(from_5d)`

这一步的结果说明：

- `PAIR(from_5d)` 优于旧 `PAIR(from_5a)`
- 说明 `5D` 作为 dense warm start，比 `5A` 更自然

所以现在角色逐渐清楚了：

- `5A`
  - 普通 actor
- `5D`
  - 普通到 dense 的桥接 actor
- `PAIR(from_5d)`
  - 当前最好的 dense 专门化 actor

### 6.2 benchmark 也要分层

为了避免后面越讲越乱，现在测试口径应该固定成三层：

- `standard_5`
  - 普通场景主测试
- `stage3_asym_three_5`
  - dense 场景主测试
- `stage2_dense`
  - 只作为压力测试，不作为当前主 benchmark

这里特别说明：

- `stage3_asym_three_5` 不是唯一 dense 场景
- 但它是当前这条主线里最合理、最稳定、最适合比较模型能力的 dense benchmark

## 7. 第四阶段：为什么 single actor overwrite 讲不通

虽然 `PAIR` 这条渐进 dense 线比旧 pure dense 顺得多，但后面又遇到另一个更本质的问题：

**继续训练单一 actor，会退化。**

我们后面做过很多版本：

- `PAIR -> THREE_5`
- `THREE_MID`
- joint-action critic 版本
- attention critic 版本
- 更晚解冻、更少更新、更保守学习率等

反复出现的现象都很像：

- warm start 模型本身还能跑
- 但 actor 一继续更新，就容易掉
- 有时普通场景能力先掉
- 有时 dense 场景也没有继续变强

所以这里得到的核心判断是：

**继续覆盖式训练同一个 actor，并不是纯增益。**

也就是说，普通导航能力和 dense 协同能力之间，至少存在明显的训练冲突。

## 8. 第五阶段：为什么单独拆出 critic 核查

对应目录：

- `03_方法核查/`

当第三课程卡住后，主线曾经怀疑是不是 critic 根本没改对。

所以又拆出一条技术核查线，重点查了：

- 集中式 critic 输入是否完整
- replay buffer 是否把多车信息存全
- 其他智能体 actor 分支处理是否合理
- detach、联合动作、局部几何这些实现细节有没有问题

这条线的结论是：

- critic 改完整后，前期确实更好
- 但长期退化问题还在

所以最后的判断变成：

**critic 不是没用，但它不是当前主矛盾。**

当前主矛盾还是：

**actor 在新阶段继续更新时的稳定性问题。**

## 9. 第六阶段：为什么现在转到“保留专门化”

对应目录：

- `04_保留专门化/`

到这里，主线判断已经比较清楚：

1. pure dense 五车旧主线不顺
2. `PAIR / THREE_5` 这种渐进 dense 课程更合理
3. `5D -> PAIR` 是目前最健康的 dense 专门化链路
4. 但继续把同一个 actor 往更复杂 dense 场景硬训，仍然容易退化

所以现在最自然的思路不再是：

- 继续硬改同一个 actor

而是：

- 保留已经学好的普通 actor
- 保留 bridge actor
- 单独准备 dense 专门化 actor
- 然后只在强交互局部状态下做增强，而不是全局重写整个策略

这就是“保留专门化”现在的来源。

## 10. 目前主线该怎么讲

现在最顺的故事，不是“我们发明了两个 actor 切换”。

更顺的讲法是：

**我们先证明了直接继续训练单一 actor 不稳定；然后发现不同阶段得到的 actor 确实有不同能力偏向；因此不再追求把同一个 actor 硬训成全能，而是转向 preserve-and-specialize。**

具体角色是：

- `5A`
  - 保留普通导航能力
- `5D`
  - 普通到 dense 的桥接基线
- `PAIR(from_5d)`
  - 当前 dense 专门化基线

## 11. 当前真正优先的方向

最近导师给的反馈很关键，意思可以收成三句：

1. 注意力可以试，但不一定是主要提升来源
2. 门控网络值得做
3. 更稳的方案是：冻结两个 actor，只单独训练门控网络

这直接影响了当前主线优先级：

- 不是先赌 attention
- 而是先做 gating

也就是说，接下来主线不该再讲成：

- “先给网络加个注意力头看看”

而应该讲成：

**保留普通 actor 和 dense actor，不再继续硬改它们，只学习一个轻量门控网络，在局部强交互时自适应选择更合适的策略。**

## 12. 现在已经做过什么

到目前为止，已经完成的大块工作可以概括成：

- 单智能体和最早的多智能体基础探索
- 正式课程学习主线
- 普通五车 actor 主线建立（`5A` / `5D`）
- 旧 pure dense 五车主线失败验证
- `PAIR(from_5a)` 和 `PAIR(from_5d)` 两版 dense actor
- `PAIR -> THREE_5` 的继续训练验证
- overwrite 路线退化验证
- critic 核查线
- 粗双 actor 切换尝试
- attention 方向的初步尝试

这些工作现在都不是孤立的，它们一起支持了当前主线判断。

## 13. 现在正在做什么

刚补完的是：

- `PAIR(from_5d)` 的正式 test
  - `standard_5`：`0.891 / 0.085 / 0.573`
  - `stage3_asym_three_5`：`0.880 / 0.122 / 0.575`

结论也很直接：

- `PAIR(from_5d)` 在标准五车上比旧 `PAIR(from_5a)` 更健康
- 但它在正式 dense benchmark 上没有超过 `5D`
- 所以它可以保留，但暂时不能直接定成门控里的 dense expert

## 14. 接下来具体要做什么

顺序应该固定，不要再乱跳：

1. 先用最稳的一组做最小门控版
   - 普通 actor：`5A`
   - dense actor：`5D`
   - 两个 actor 全冻结
   - 只训练 gating network

2. 再把 `PAIR(from_5d)` 补成对照
   - 比较 `5A + 5D` 和 `5A + PAIR(from_5d)`
   - 看 dense test 到底是谁更适合做 expert

补充一条执行层顺序：

- 在直接训练 learned gate 之前，先做：
  - `5A + 5D` 的 `hard_switch`
  - `5A + 5D` 的 `case_oracle`
- 先确认两个 actor 是否真有足够互补性，再决定要不要继续投到 gate 训练上

3. 门控输入先保持简单
   - 自车状态
   - 激光的低维摘要
   - 最近邻距离
   - 邻居数量或风险指标

4. 第一版先验证门控是否有效
   - 能不能保住 `standard_5`
   - 能不能提升 `stage3_asym_three_5`

5. 只有当门控已经有效时，再试 attention
   - 这时 attention 是增强项
   - 不是成败点

6. 如果整体有效，再考虑安全兜底或传统控制分支

## 15. 现在这一版最简短的一句话

**从单智能体到普通多车，我们已经把基础主线跑通了；旧的 pure dense 五车路线学不顺，继续硬训单一 actor 又容易退化；现在最稳的做法不是继续覆盖训练，而是保留普通 actor，再保留一个 dense actor，先从 `5A + 5D` 的冻结双 actor 门控开始，`PAIR(from_5d)` 作为后续专门化对照。**
